# LLaVA-OneVision：单图、多图、视频统一模型

> 在 LLaVA-OneVision（Li et al.，2024 年 8 月）之前，开源 VLM 世界有各自独立的血统：LLaVA-1.5 做单图，Mantis 和 VILA 等做多图，Video-LLaVA 和 Video-LLaMA 做视频。每个在自己的基准上赢，在其他基准上失败。LLaVA-OneVision 论证了单一课程学习可以训练一个模型在所有三种场景中占主导，而且涌现的任务迁移效应（单图技能导出到视频，多图推理导出到单图）超过了专家模型之和。方案看似简单：一个跨场景保持恒定的视觉 token 预算，加上一个从单图到 OneVision（多图）到视频的显式课程。本课解读预算、课程和涌现行为。

**类型：** 构建
**语言：** Python（标准库，token 预算求解器 + 课程规划器）
**前置：** Phase 12 · 05（LLaVA）、Phase 12 · 06（任意分辨率）
**时间：** ~180 分钟

## 学习目标

- 设计一个跨单图、多图和视频输入保持恒定的视觉 token 预算。
- 排列训练课程顺序，使技能从单图迁移到视频而不发生灾难性遗忘。
- 解释为什么在相同参数量下，课程做对了的单一模型能超过专家模型。
- 说出 LLaVA-OneVision 报告的三种涌现能力：多摄像头推理、set-of-mark prompting、iPhone 截图 agent。

## 问题

图像、多图和视频各自以不同方式给模型施压。

单图需要高分辨率 token（AnyRes，~2880 视觉 token）来捕捉 OCR 和细节。每样本预算：一张图，2880 token。

多图需要多张中等分辨率的图像（每张 ~576 token），使跨图推理能放进上下文。每样本预算：4-8 张图，每张 576，2300-4600 token。

视频需要多帧低分辨率（池化后每帧 ~196 token）来捕捉时间动态。每样本预算：8-32 帧，每帧 196，1600-6200 token。

如果训练独立模型，你选一个预算。如果训练一个模型，你需要预算在各场景间合理缩放而不撑爆上下文。

OneVision 之前，默认答案是"训练一个场景，忽略其他"。Video-LLaVA 用额外训练阶段将视频改装到图像模型上。LLaVA-NeXT 用 tiling 添加了多图支持。没有一个能干净地处理全部三种。

## 概念

### OneVision token 预算

LLaVA-OneVision 选择了约 3000-4000 token 每样本的统一视觉 token 预算，按场景不同分配：

- 单图：AnyRes-9（3x3 tile + 缩略图），每个 tile 在 384 分辨率有 729 个 patch，激进的 2x2 双线性池化 → 每 tile 182。总计：9 * 182 + 182 = 1820 token。或 AnyRes-4 在 729-per-tile = 2916 + 729。
- 多图：每张图在中等分辨率（384，无 tiling），729 token 无池化。预算 6 张图 → 4374 token。
- 视频：32 帧在 384 分辨率加激进 3x3 双线性池化 → 每帧 81 token。总计：32 * 81 = 2592 token。

分配保持总 token 大致恒定。LLM 永远不会看到撑爆上下文的 batch。编码器按场景产生不同几何，但 LLM 消费相同预算。

### 三阶段课程

LLaVA-OneVision 分三个阶段训练：

1. 单图 SFT（阶段 SI）。所有数据是单图加文本。在高分辨率 AnyRes 输入上训练。教会感知、OCR 和细粒度理解。使用 LLaVA-NeXT 数据加 OneVision 特定单图数据。
2. OneVision SFT（阶段 OV）。混合单图 + 多图 + 视频（均匀采样帧）。在统一 token 预算上训练。教会模型处理异构 batch 形状。不重置权重——从阶段 SI 继续。
3. 任务迁移（阶段 TT）。继续用目标任务混合，通常偏重多图或视频取决于产品。可选的部署微调。

关键：课程顺序很重要。先训视频或先训多图比先训单图产生更差的图像性能，即使数据相同。论文明确消融了这一点。

### 为什么课程有效

单图训练建立感知基础。Patch token 携带细粒度视觉特征；LLM 学会将它们与文本整合。多图和视频引入结构性挑战（哪张图是哪张，什么先发生），这些在没有强感知基础的情况下很难学。

如果从头一起训练所有场景，模型对感知欠拟合（每 batch 单图数据有限）而对结构过拟合（大量多图/视频数据）。结果：一个遵循跨图推理模式但视觉浅薄的模型。

课程排序给你阶段 SI 的感知强度，然后阶段 OV 的组合/时间推理，两者都不丢失。

### 涌现的跨场景技能

LLaVA-OneVision 论文报告了三种涌现能力：

1. 多摄像头推理。分别在多图 + 视频上训练；推理时被要求推理多摄像头驾驶场景。模型正确整合了视角，尽管训练中从未见过那种确切格式。
2. Set-of-mark prompting。用户在图像中用编号标记标注物体；模型推理"标记 3 相对于标记 7 在做什么"。既没有在标记上训练也没有在标注上训练；从空间定位 + 多图参考的组合中学到。
3. iPhone 截图 agent。用户提供 iPhone 屏幕截图并要求规划下一次点击。在 UI 截图、用户工作流视频和多图前后对比对上训练。泛化到 agent 用例。

这些不是训练任务；它们从课程的组合结构中涌现。

### 视觉 token 池化

Token 预算需要池化。OneVision 在 2D patch 网格上使用双线性插值：24x24 = 576 个 patch 变成 12x12 = 144（2x 因子）或 8x8 = 64（3x 因子）。池化在 patch 网格空间而非 token 空间中进行，以保持局部性。

每个场景的池化因子本身是一个超参数。更少池化 = 更多 token = 更丰富的表示。更多池化 = 更少 token = 更多帧/图像能放进去。

### LLaVA-OneVision-1.5

2025 年的后续（LLaVA-OneVision-1.5，arXiv 2509.23661）在训练数据、模型权重和代码上"完全开放"。在某些基准上匹配专有差距并民主化了方案。相同课程，更多数据，更好的基础 LLM。无架构变化。

### 与 Qwen2.5-VL 的对比

Qwen2.5-VL（课程 12.09）做了不同选择。它使用 M-RoPE 和动态 FPS 而非固定池化。它的预算随输入缩放——1 分钟视频比 5 秒视频用更多 token。LLaVA-OneVision 固定预算并缩放池化。两者都有效；它们在可配置性和可预测性之间权衡。

## 动手用

`code/main.py` 是一个 OneVision 风格 VLM 的课程和预算规划器。给定每样本 token 预算和目标场景混合（比如 40% 单图、30% 多图、30% 视频），它：

- 按场景分配分辨率、池化因子和帧数。
- 检查每个场景是否在共享预算内。
- 报告预期 token 数、LLM FLOPs，以及哪些场景 token 不足。
- 打印逐阶段训练计划。

用它来规划 OneVision 微调或检查 VLM 部署的每请求成本。

## 交付物

本课产出 `outputs/skill-onevision-budget-planner.md`。给定目标任务分布和每样本预算，它输出 AnyRes 因子、每帧池化、视频帧数和课程阶段权重。每当你训练或微调统一场景 VLM 时使用。

## 练习

1. 你的产品支持 80% 单图、10% 多图（2-4 张）、10% 视频（8-16 帧）。设计 token 预算。你从不做重度多图中省下的额外预算放在哪里？

2. 阅读 LLaVA-OneVision 第 4.3 节（涌现能力）。提出一个课程可能解锁但论文未报告的第四种涌现技能。

3. 交换课程顺序——先训多图，然后单图，然后视频。预测哪些基准下降以及为什么。

4. 论文报告视频基准在每样本仅 8 帧上训练。这能泛化到推理时的 30 秒视频吗？什么先崩溃——token 预算还是时间推理？

5. 24x24 patch 到 12x12 的双线性池化是每维 4x 缩减。用标准库 Python 实现池化并验证每个 2x2 块的均值与双线性输出匹配。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| OneVision scenario | "单图、多图或视频" | 统一 VLM 处理的三种输入形状之一；预算跨场景保持恒定 |
| Token budget | "每样本多少 token" | LLM 每训练/推理样本看到的总视觉 token，通常 3000-4000 |
| Curriculum | "训练顺序" | 为涌现迁移选择的阶段排序（单图 → 多图 → 视频） |
| Bilinear pooling | "Token 缩减" | 对 patch 网格（2D）应用双线性插值以减少 token 数同时保持局部性 |
| Emergent skill | "没训过，但能用" | 推理时出现的能力，没有匹配的训练数据，源于课程组合 |
| AnyRes-k | "k-tile 设置" | k 个固定分辨率子 tile 加一个缩略图，典型 k ∈ {4, 9} |
| Task transfer | "跨场景泛化" | 在单图上学到的技能通过共享骨干应用到视频（反之亦然） |
