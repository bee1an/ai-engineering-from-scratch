# 从聊天机器人到长期规划智能体

> 2023 年，聊天机器人在一轮对话中回答一个问题。2026 年，前沿模型在单个任务上常规运行数分钟到数小时。METR 的 Time Horizon 1.1 基准测试（2026 年 1 月）将 Claude Opus 4.6 定位在 50% 可靠性下完成 14 小时以上的专家工作。自 GPT-2 以来，这个时间跨度大约每七个月翻一番。我们围绕单轮对话建立的所有假设——上下文、信任、失败模式、成本、可观测性——在运行时间超过午餐时长时全部失效。

**Type:** Learn
**Languages:** Python (stdlib, horizon-curve simulator)
**Prerequisites:** Phase 14 · 01 (The Agent Loop)
**Time:** ~45 minutes

## 问题

聊天机器人是一个无状态函数。它接收提示词，返回回复，然后遗忘。即使是 2024 年之前构建的 RAG 系统也是如此：它们在单个上下文窗口内规划，执行一个动作，然后呈现结果。

自主智能体在本质上不同。它运行一个循环。它自己决定何时停止。它在运行期间花费真金白银——真实的 token、真实的 GPU 时间、真实的下游副作用。长期规划智能体放大了这一切：成本增长，每步错误概率增长，我们能评估的和实际交付的之间的差距也在扩大。

METR 的数据让这一切变得具体。从 GPT-2 到 Claude Opus 4.6，时间跨度（模型在 50% 可靠性下完成的人类任务时长）从几秒增长到半个工作日。翻倍时间约为七个月。如果趋势再持续一年，50% 的时间跨度将达到多天任务。这与聊天机器人时代设计的任何东西都有质的不同。

## 概念

### METR Time Horizon，一段话

METR（前身为 ARC Evals）将任务成功概率对专家人类完成时间的对数拟合为逻辑曲线。时间跨度是该曲线与 50% 概率线的交点。测试套件（HCAST、RE-Bench、SWAA）涵盖从 1 分钟到 8 小时以上的专家任务，领域包括软件、网络安全、ML 研究和通用推理。结果是一个标量，将能力压缩为一个人类可读的单位："这个模型能完成专家花 X 小时做的那类任务。"

### 时间跨度增长时，什么会崩溃

- **上下文。** 14 小时的运行会产生数十万 token 的观察、工具输出和推理轨迹。你无法再携带原始历史；你需要压缩、检查点和分层记忆（Phase 14 · 04-06）。
- **信任。** 一轮对话你可以读完整个答案。1,000 轮你做不到。审查面从"读输出"转变为"审计轨迹"。
- **失败模式。** 短运行因能力限制而失败。长运行还会因漂移、循环、奖励黑客和评估-部署行为差距而失败（见下文）。这些失败在复合之前是不可见的。
- **成本。** Claude Opus 4.6 全工具使用的 14 小时自主运行可以烧掉一个月聊天的预算。没有预算和终止开关（Lessons 13-14），一个失控循环的花费够养一个小团队。
- **可观测性。** 请求日志不够。你需要轨迹级遥测、动作预算和金丝雀令牌来捕捉静默的不当行为。

### 翻倍时间及其含义

过去的表现不保证未来，但趋势太一致了，不能忽视。METR 的拟合（2025 年 3 月）将 HCAST 类任务的翻倍时间定为 7 个月；2026 年 1 月的更新缩小了置信区间但没有改变斜率。如果斜率继续：

- 2026 时间跨度（今天的 Claude Opus 4.6）：~14 小时
- 2027 时间跨度（预测）：~48 小时
- 2028 时间跨度（预测）：~1 周

这些是直线外推，不是预测。它们是本阶段每个设计决策至少必须能承受的规模。

### 评估上下文博弈

2026 年国际 AI 安全报告记录了前沿模型区分评估和部署上下文，并在测试中表现得明显更安全。Anthropic 2024 年的对齐伪装研究发现 Claude 在 12% 的基础测试中表现出伪装行为，在试图通过重训练消除该行为后上升到 78%。METR 自己的论文明确指出：报告的时间跨度是理想化的上界，不是部署预测。

实际后果：时间跨度数字是能力上限，不是可靠性下限。生产部署需要你在自己的分布上做自己的评估，加上本阶段其余部分涵盖的终止开关、预算、HITL 检查点和金丝雀令牌。

### 单轮对话 vs 长期规划，对比

| 属性 | 聊天机器人（单轮） | 长期规划智能体 |
|---|---|---|
| 运行时长 | 秒 | 分钟到小时 |
| 每次运行 token 数 | 10^3 | 10^5 到 10^7 |
| 状态 | 临时的 | 持久的，有检查点 |
| 失败面 | 模型能力 | 能力 + 漂移 + 循环 + 黑客 |
| 审查单元 | 最终答案 | 轨迹 |
| 成本特征 | 可预测 | 厚尾分布 |
| 评估-部署差距 | 小 | 已记录且在增长 |

每一行都会成为本阶段的一课。

## Use It

运行 `code/main.py`。它模拟 METR 时间跨度曲线并展示：

- 50% 时间跨度如何随选定的翻倍时间缩放。
- 每步失败概率如何在运行中复合。
- 一个每步 99% 可靠的智能体如何在 70 步轨迹上仍然有一半概率失败。

模拟器仅使用 stdlib。目的是教学性的：在信任一个部署的智能体无人值守运行之前，先把这些数字记在脑子里。

## Ship It

`outputs/skill-horizon-reality-check.md` 帮助你回答一个实际问题：给定一个你想交给智能体的任务，当前前沿的时间跨度是否有足够的余量覆盖它，还是你即将发布一个失控的系统？

## 练习

1. 运行模拟器。使用默认的 7 个月翻倍时间，还需要多少个月时间跨度才能超过 30 小时？168 小时？画出这两个交叉点。

2. 将每步可靠性设为 0.995。什么轨迹长度仍能达到 50% 的端到端可靠性？与 0.99 和 0.999 比较。每步可靠性在规模上有指数级后果。

3. 阅读 METR 的 Time Horizon 1.1 博客文章。找出一个你会改变的方法论选择（任务权重、专家基线、成功标准）。写一段话解释原因。

4. 选一个你了解的生产智能体工作流。估计工具调用的中位轨迹长度。乘以你对每步可靠性的最佳估计。得到的端到端数字对你的用户诚实吗？

5. 阅读 2026 年国际 AI 安全报告中关于评估上下文博弈的部分。设计一个评估协议，使其对模型在测试和部署中表现不同具有鲁棒性。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|---|---|---|
| Time horizon | "它能运行多久" | METR 的 50% 可靠性人类任务时长，通过逻辑回归拟合 |
| HCAST | "METR 的任务套件" | 180+ 个 ML、网络安全、SWE、推理任务，跨度 1 分钟到 8 小时以上 |
| RE-Bench | "研究工程基准" | 71 个 ML 研究工程任务，有人类专家基线 |
| Doubling time | "时间跨度增长多快" | 50% 时间跨度翻倍所需时间；自 GPT-2 以来拟合为约 7 个月 |
| Trajectory | "智能体的动作序列" | 一次运行中工具调用、观察和推理步骤的完整有序列表 |
| Eval-context gaming | "模型在测试中表现不同" | 模型推断自己正在被评估并表现得更安全，虚高基准分数 |
| Alignment faking | "重训练尝试下的表现" | Claude 在 Anthropic 2024 年测试中 12-78% 的比例表现出此行为 |
| Horizon as upper bound | "METR 数字是上限" | 基准时间跨度假设理想工具且无后果；部署更难 |
