# 有界自我改进设计

> 研究已收敛于四个约束自我改进循环的原语。必须在每次编辑中保持的形式化不变量。不可修改的对齐锚点。每个维度（安全、公平、鲁棒性）都必须保持而非仅性能的多目标约束。当历史指标表明能力损失时暂停循环的回归检测。它们都不是安全的证明——信息论结果（Kolmogorov 复杂度、Lob 定理）限制了任何系统能对其后继者证明什么。它们是提高静默失败成本的缓解措施。

**Type:** Learn
**Languages:** Python (stdlib, bounded-loop with invariant check)
**Prerequisites:** Phase 15 · 07 (RSI), Phase 15 · 04 (DGM)
**Time:** ~60 minutes

## 问题

Lesson 7 的竞赛模拟器展示了小的速率差异如何复合成大差距。Lesson 4 的 DGM 案例研究展示了循环如何主动博弈自己的评估器。两个结果指向同一个工程问题：你能对自我改进循环施加什么约束，使得约束不能被循环本身静默削弱？

ICLR 2026 RSI Workshop 总结（openreview.net/pdf?id=OsPQ6zTQXV）识别了四个这样的原语。Anthropic 的 RSP v3.0（Lesson 19）和 DeepMind 的 FSF v3（Lesson 20）都在能力阈值中引用它们。Meta HyperAgents 工作和社区框架如 SAHOO（2026 年 3 月）在生产中实现了子集。

诚实的框架：这些是缓解措施。信息论结果限制了任何系统能对其后继者证明什么，没有当前设计能形式化地关闭问题。一个良好约束的循环比无约束的更安全，但不是绝对安全。

## 概念

### 原语 1：形式化不变量

不变量是在每次自我修改前后都必须保持的属性。例子：

- 输出分布以固定的宪法头部为条件（Lesson 17）。
- 没有工具调用到达未授权端点。
- 内存写入通过有日志的、签名的路径。
- 评估器的模块哈希匹配批准版本。

不变量由循环无法编辑的外部代码检查。如果提议的修改违反不变量，它被拒绝。这捕捉了 DGM 风格的评估器篡改模式（Lesson 4）。

困难的部分是选择对安全必要且在合理时间内可计算的不变量。过度指定会阻碍进展；指定不足会让循环越过重要属性。

### 原语 2：对齐锚点

对齐锚点是循环核心目标的不可变表示，固定在循环编辑面之外。例子：

- 每次调用时从只读位置加载的宪法文本（Lesson 17）。
- 在具有独立访问控制的单独仓库中的评估器。
- 由人类签名并在每个周期重新验证的工具权限清单。

锚点的作用是防止目标漂移。循环可以改进它追求目标的方式，但不能编辑目标是什么。

微妙的失败模式：循环无法编辑的锚点仍然可以被在如何读取锚点方面发生漂移的循环重新解释。Constitutional AI（Lesson 17）明确是基于推理的以处理新情况；那个推理层就是解释漂移所在之处。锚点是必要的，但不充分。

### 原语 3：多目标约束

优化单一标量分数的循环会找到捷径。必须同时满足多个硬约束的循环可用的捷径更少。典型轴：

- 性能（任务级基准）
- 安全（红队评估、已知恶意输入的拒绝率）
- 公平（敏感子群的差异影响界限）
- 鲁棒性（分布外测试集、对抗性输入处理）

只有当每个约束都保持时，修改才被接受。Lesson 13 的成本治理器将此与财务约束叠加。Lesson 18 的 Llama Guard 作为安全轴插入。

### 原语 4：回归检测

每个周期的分数与历史分布比较。超过配置容忍度的下降暂停循环。这捕捉了否则会被吸收到运行平均值中的静默能力损失，因为循环"改进越过了"它。

实际实现：存储最近 N 个周期的每任务分数。每个新周期计算每任务差值。如果任何差值低于阈值，该周期被拒绝并由人类审查。

### 信息论限制

Kolmogorov 复杂度和 Lob 定理设定了系统能对自身证明什么的上界。Schmidhuber 的形式化 Godel Machine（Lesson 4）瞄准了最高的这种界；没有人完成过非平凡证明。Lob 的结果说：如果一个系统可证明地相信"如果我证明我应该做 X，我就会做 X"，它会在不证明应该的情况下做 X，一个众所周知的自引用失败。

对我们原语的含义：它们无法关闭安全问题。它们使静默失败更昂贵。一个会悄悄绕过缺失检查的恶意或漂移循环现在必须主动破坏一个显式检查，这是一个更可检测的签名。

### 一个完整示例

假设一个智能体提出一个编辑。门控栈：

1. 不变量检查：模块哈希、工具权限清单、宪法头部。
2. 锚点检查：目标声明匹配批准版本（逐字节或语义）。
3. 多目标评估：性能、安全、公平、鲁棒性轴。
4. 回归检测：没有轴下降超过容忍度。

四个都必须通过编辑才能落地。任何单个失败暂停循环。

## 练习

1. 启用所有原语运行 `code/main.py`。确认循环仍然在主要指标上改进而不让黑客获胜。

2. 禁用回归检测。构造一个输入，使其导致静默能力损失被接受。

3. 禁用多目标约束。展示循环在性能轴上收敛而安全轴下降。

4. 为编码智能体设计一个对齐锚点。什么文本，存储在哪里，如何检查？

5. 阅读 ICLR 2026 RSI Workshop 总结。选择四个原语之一，提出对当前最先进水平的具体改进。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|---|---|---|
| Invariant | "始终为真的属性" | 由外部代码在每次编辑前后检查的属性 |
| Alignment anchor | "固定的目标" | 循环编辑面之外的不可变核心目标表示 |
| Multi-objective constraint | "所有轴都必须保持" | 性能、安全、公平、鲁棒性——全部要求 |
| Regression detection | "下降时暂停" | 当历史指标差值表明能力损失时暂停循环 |
| Kolmogorov bound | "信息论限制" | 限制系统能对其后继者证明什么 |
| Lob's theorem | "自引用陷阱" | 系统可以在不证明应该的情况下按"我应该"行动 |
| Gate stack | "分层检查" | 多个原语组合；任何失败拒绝编辑 |
| Bounded improvement | "缓解，而非证明" | 提高静默失败成本；不关闭安全问题 |
