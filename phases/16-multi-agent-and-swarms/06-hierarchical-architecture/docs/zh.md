# 层级架构及其失败模式

> 层级就是嵌套的监督者。管理者 agent 管理子管理者，子管理者管理工作者。CrewAI `Process.hierarchical` 是教科书版本：一个 `manager_llm` 动态委派任务并验证输出。LangGraph 的等价物是 `create_supervisor(create_supervisor(...))`。当任务是真正的组织架构图时，这是自然的模式。它也是最容易崩溃为管理循环的模式——管理者 agent 分配工作不当、误解子输出、或无法达成共识。顺序模式往往表现更好。

**Type:** Learn + Build
**Languages:** Python (stdlib)
**Prerequisites:** Phase 16 · 05 (Supervisor Pattern)
**Time:** ~60 minutes

## 问题

一旦监督者模式理解了，自然的下一步是"如果工作者本身也是监督者呢？"团队有子团队；公司有部门的部门。层级架构映射了这一点。

问题是：LLM 管理者不同于人类管理者。人类管理者对其下属知道什么有稳定的先验。LLM 管理者每一轮都从上下文中的内容重新推理组织。上下文中的微小漂移，整棵树就会错误分配工作。

## 概念

### 形状

```
                 Manager
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       Sub-Mgr A         Sub-Mgr B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点规划、委派和综合。只有叶子做工作。

### 适用场景

- **清晰的组织映射。** 如果真实任务是部门化的（"法务审查文档、财务审查文档、工程审查文档，然后为高管总结"），层级是显式的。
- **局部摘要。** 每个子管理者在顶层管理者看到之前综合其团队的输出。顶层管理者看到三个子管理者摘要，而不是十五个工作者输出。

### 失败场景

2026 年事后分析不断发现的三种失败模式：

1. **任务分配错误。** 管理者读取目标，幻觉出一个分解，委派给错误的子管理者。因为子管理者顺从地处理被分配的内容，错误只在顶层综合时才浮现——比人类能捕获的位置晚了一层。
2. **输出误解。** 子管理者返回"无法验证声明 X"。顶层管理者总结为"声明 X 未确认"。含义在每一层漂移。
3. **共识循环。** 两个子管理者不同意；顶层管理者要求他们协调；他们向下重新委派；工作者重新运行；子管理者返回略有不同的答案；循环。CrewAI 的 `Process.hierarchical` 用步骤限制防护，但限制本身现在是一个超参数。

### 决定性问题

顺序（线性流水线）vs 层级：你的任务实际上有独立的子团队，还是一个伪装成树的线性流程？如果是后者，用顺序。如果是前者，用层级但预算显式的协调规则。

### CrewAI 的实现

`Process.hierarchical` 在专家 crew 之上连接一个管理者 LLM。管理者：

- 接收顶层任务，
- 将子任务分配给 crew，
- 评估 crew 输出，
- 决定是接受、重新委派还是迭代。

文档：https://docs.crewai.com/en/introduction（在 Core Concepts 下查找"Hierarchical Process"）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内部监督者有自己的图；外部监督者将内部图视为不透明节点。这比 CrewAI 更容易调试（你可以分别步进每个图），但更难表达树的动态重塑。

参考：https://reference.langchain.com/python/langgraph-supervisor。

## Build It

`code/main.py` 运行一个 3 层层级：

- 顶层管理者：将任务拆分为"工程"和"法务"分支，
- 工程子管理者：拆分为"前端"和"后端"工作者，
- 法务子管理者：一个工作者。

演示对比正常路径（所有人同意）和一个**扰动路径**，其中顶层管理者的分解将"法务"错误标记为"财务"，并观察错误级联——子管理者顺从地做财务工作，顶层综合器报告财务发现，原始法务问题无人回答。

运行：

```
python3 code/main.py
```

输出显示两条路径，清晰地并排对比"被问了什么"vs"交付了什么"。

## Use It

`outputs/skill-hierarchy-fitness.md` 评估给定任务应该使用层级、顺序还是扁平监督者。输入：任务描述、组织结构、协调预算。输出：模式推荐及需要防护的具体失败模式。

## Ship It

如果你部署层级：

- **树深度上限为 2。** 三层已经将大多数错误隐藏在可观测性之外。
- **显式协调预算。** 设置顶层管理者必须提交前的最大轮数。通常 2。
- **每次综合的溯源。** 每个节点的摘要必须引用哪些叶子输出产生了它。
- **分解漂移告警。** 记录管理者每步的分解；与用户查询 diff。如果分解不再覆盖查询，触发告警。

## 练习

1. 运行 `code/main.py` 并比较正常 vs 扰动。需要多少层管理者传递才能使顶层输出完全偏离用户的问题？
2. 添加第三层（top → sub → sub-sub → worker）。测量随着深度增长，扰动路径自我纠正 vs 完全偏离的频率。
3. 在每个子管理者处实现一个"金丝雀"工作者，始终被问原始用户问题不变。使用金丝雀答案检测分解漂移。当金丝雀与综合答案不一致时，管理者应如何反应？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。识别 CrewAI 应用的一个具体防护措施（步骤限制、manager_llm 约束）并描述它针对什么失败模式。
5. 比较嵌套 LangGraph 监督者与 CrewAI 层级。哪个使协调循环更容易检测？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|---------|---------|
| 层级 | "组织架构图模式" | 监督者之上的监督者；只有叶子做工作。 |
| 管理者 LLM | "老板" | 在内部节点分解、分配和验证的 LLM。 |
| 分解漂移 | "老板跑偏了" | 顶层管理者的拆分不再覆盖原始问题。 |
| 协调循环 | "无尽的会议" | 子管理者不同意；顶层重新委派；工作者重新运行；循环直到预算耗尽。 |
| 深度-2 上限 | "不要超过 2 层" | 经验防护：3+ 层崩溃可观测性。 |
| 金丝雀问题 | "每层的真相" | 一个始终被问原始查询不变的工作者，用于检测漂移。 |
| 溯源链 | "谁说了什么" | 从每次综合追溯到产生它的叶子输出。 |
