# 角色专业化 — 规划者、批评者、执行者、验证者

> 2026 年最常见的多 agent 分解：一个 agent 规划，一个执行，一个批评或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示中的 SOP——产品经理、架构师、项目经理、工程师、QA 工程师——遵循 `Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过"聊天链"串联设计师、程序员、审查员、测试员，并使用"沟通去幻觉"（agent 显式请求缺失细节）。验证者是承重的：Cemri 等人（MAST，arXiv:2503.13657）表明每个多 agent 失败都可以追溯到缺失或损坏的验证。PwC 报告在 CrewAI 中通过结构化验证循环获得了 7 倍准确率提升（10% → 70%）。

**Type:** Learn + Build
**Languages:** Python (stdlib)
**Prerequisites:** Phase 16 · 04 (Primitive Model), Phase 16 · 05 (Supervisor)
**Time:** ~60 minutes

## 问题

通用多 agent 系统产出通用输出。群聊中的三个编码器写出三种风味的同样平庸代码。你可以添加更多 agent、更多轮次，仍然无法跨越质量门槛。

修复方法不是更多 agent——而是*不同的* agent。分配不同角色。给批评者规划者没有的工具。给验证者一个客观的测试套件。现在系统有了基于事实的内部分歧和纠正，而不只是并行猜测。

## 概念

### 四个规范角色

**规划者。** 读取目标，产出步骤列表或规格。工具：知识检索、文档。输出：结构化计划。

**执行者。** 一次读取一个计划步骤，产出制品。工具：实际工作工具（代码编译器、shell、API 客户端）。输出：制品。

**批评者。** 对照规划者的意图读取执行者的输出。工具：对制品的只读访问、静态分析。输出：接受/拒绝及理由。

**验证者。** 读取制品并运行确定性检查。工具：测试运行器、类型检查器、schema 验证器。输出：通过/失败及证据。

批评者是主观的、有观点的，通常基于 LLM。验证者是客观的、确定性的，通常基于代码。它们不是同一个角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程 SOP 编码为角色提示：

- **产品经理**写 PRD。
- **架构师**产出系统设计。
- **项目经理**拆分任务。
- **工程师**实现。
- **QA 工程师**运行测试。

每个角色有严格的输入/输出 schema。角色提示说明角色*是什么*以及它*必须产出什么*。`Code = SOP(Team)` 公式——确定性 SOP 将一个 LLM 团队变成可预测的流水线。

### ChatDev 的沟通去幻觉

ChatDev 添加了一个关键动作：当执行者需要计划中没有的具体细节时，它在继续之前显式询问设计师。这防止了 LLM 合理地编造细节的经典失败。

实现：角色提示包含"当你需要未被给予的具体信息时，在产出输出之前按名称询问相关角色。"

### 为什么验证者最重要

Cemri 等人（MAST）追踪了 1642 个多 agent 执行失败。21.3% 是验证缺口——系统发布了一个没人检查过的答案。剩余 79% 通常追溯到"有一个检查静默失败或从未运行"。验证是承重角色。

PwC 报告（CrewAI 部署，2025）添加结构化验证循环将准确率从 10% 提升到 70%。一个角色带来 7 倍提升。

### 批评者 vs 验证者

- 批评者是一个 LLM 审查制品质量。主观。可被合理的文字欺骗。
- 验证者是一个在制品上运行的确定性程序。客观。给出通过/失败及证据。

两者都用。批评者捕获验证者无法表达的品味问题。验证者捕获批评者看不到的 bug，因为它们只在运行时才显现。

### 反模式

你系统中的每个角色都是 LLM，每个角色的输出都是"看起来不错"。经典 MAST 失败模式。添加至少一个验证者，其通过/失败由代码决定，而不是 LLM。

### 框架映射

- **CrewAI** — `Agent(role, goal, backstory)` 是教科书级的专业化表面。
- **LangGraph** — 节点可以有专业化提示；边强制流水线。
- **AutoGen** — GroupChat 中带单词名称的角色特定 ConversableAgent。
- **OpenAI Agents SDK** — 角色专业化 Agent 之间的 handoff 工具。

## Build It

`code/main.py` 实现了一个构建简单 Python 函数的 4 角色流水线：

- **规划者**产出规格。
- **执行者**生成代码字符串。
- **批评者**（LLM 模拟）标记明显问题。
- **验证者**在沙箱（`exec`）中对测试用例运行生成的代码。

演示运行两次：一次执行者产出正确代码（批评者 + 验证者都通过），一次执行者产出不符合规格的代码（批评者因为看起来合理而漏掉 bug，验证者因为测试失败而捕获它）。

运行：

```
python3 code/main.py
```

## Use It

`outputs/skill-role-designer.md` 接受任务并产出角色名单（3-5 个角色）、每个角色的输入/输出 schema 和验证者检查。在将 agent 连接到框架之前使用。

## Ship It

检查清单：

- **至少一个确定性验证者。** 永远不要全 LLM。
- **每个角色的显式 I/O schema。** 规划者返回规格，不是散文；执行者读取该 schema。
- **沟通去幻觉。** 执行者在信息缺失时必须询问规划者；永远不要编造。
- **批评者/验证者顺序。** 先运行批评者（便宜，捕获设计问题），再运行验证者（慢，捕获 bug）。
- **循环预算。** 最多 2 轮批评者-执行者修订循环，然后升级到人类。

## 练习

1. 运行 `code/main.py` 并观察验证者如何捕获批评者漏掉的 bug。添加一个静态分析检查（计算 `return` 出现次数）作为额外验证者。它捕获了什么运行时测试漏掉的？
2. 添加第 5 个角色："需求分析师"，将用户愿望翻译为规划者就绪的规格。什么沟通去幻觉请求应该向上流向它？
3. 阅读 MetaGPT 第 3 节（"Agents"）。列出 MetaGPT 5 个角色各自的输入/输出 schema。
4. 阅读 ChatDev 的聊天链图（arXiv:2307.07924 Figure 3）。识别沟通去幻觉在哪里打断了否则会无限的循环。
5. PwC 的 7 倍准确率提升来自验证循环。假设三个添加验证者不会有帮助的任务——确定性正确性检查不可能或成本过高的场景。

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|---------|---------|
| 角色专业化 | "不同 agent，不同工作" | 为规划者/执行者/批评者/验证者角色调优的不同系统提示。 |
| SOP 模式 | "编码的标准操作程序" | MetaGPT 的框架：严格的 I/O schema 将团队变成流水线。 |
| 沟通去幻觉 | "问了再说" | ChatDev 模式：执行者在细节缺失时询问规划者而不是编造。 |
| 批评者 | "LLM 审查员" | 主观的、有观点的审查员。捕获品味问题。可被合理文字欺骗。 |
| 验证者 | "确定性检查" | 基于代码的通过/失败。测试运行器、类型检查器、schema 验证器。不可被欺骗。 |
| 验证缺口 | "没人检查" | 21.3% 的 MAST 失败。答案在没有本可捕获 bug 的检查的情况下发布。 |
| 修订循环 | "批评者打回去" | 批评者拒绝触发执行者带反馈重新运行。需要预算。 |
| 全 LLM 反模式 | "看起来不错" | 每个角色都是 LLM，没有确定性检查。经典 MAST 失败。 |
