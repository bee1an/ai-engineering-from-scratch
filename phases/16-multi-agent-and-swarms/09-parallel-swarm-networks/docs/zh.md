# 并行 / Swarm / 网络化架构

> 与监督者对比：没有中央决策者。Agent 读取共享事件总线，异步领取工作，写回结果。LangGraph 明确支持"Swarm Architecture"用于去中心化、动态环境。Matrix（arXiv:2511.21686）将控制流和数据流都表示为通过分布式队列传递的序列化消息，以消除编排器瓶颈。权衡是显式的：用确定性和可追踪性换取可扩展性。Swarm 适合有许多独立子问题的任务；不适合需要单一连贯计划的任务。

**Type:** Learn + Build
**Languages:** Python (stdlib, `threading`, `queue`)
**Prerequisites:** Phase 16 · 05 (Supervisor Pattern), Phase 16 · 04 (Primitive Model)
**Time:** ~75 minutes

## 问题

监督者可以扩展到几个工作者。几百个呢？监督者本身成为瓶颈：关于谁做什么的每个决策都通过一个 agent。一个慢的计划步骤拖住整个系统。

Swarm 架构翻转了设计。不是中央规划者分派工作，而是工作者从共享队列中领取工作。"协调"被烘焙到事件总线语义中。没有编排器；系统扩展直到队列成为瓶颈。

## 概念

### 形状

```
                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool
```

没有编排器。每个工作者重复：拉取任务、处理、写入结果（并可选地入队后续任务）。

### Swarm 适用场景

- **许多独立任务。** 爬取、转换、分类。任务之间不相互依赖。
- **可变时长工作。** 如果有些任务需要 100ms 而其他需要 10s，swarm 自动平衡负载——快的工作者拉取下一个任务。监督者必须预估时长。
- **吞吐量优先于确定性。** 你关心总完成时间，而不是严格顺序。

### Swarm 失败场景

- **有序工作流。** 如果步骤 3 需要步骤 2 的输出，swarm 有步骤 3 在步骤 2 完成前触发的风险。
- **全局计划任务。** 复杂研究问题受益于规划者。一群研究者产出独立事实，而不是连贯报告。
- **调试。** 没有中央日志和异步工作，重现 bug 很昂贵。

### Matrix（arXiv:2511.21686）

Matrix 是 2025 年的论文，将 swarm 推到其自然结论：控制流和数据流都是分布式队列上的序列化消息。没有中央协调者。容错来自消息持久性。可扩展性是消息代理的问题，不是系统的。

贡献：一个编程模型，其中多 agent 协调是"这个 agent 订阅什么消息主题？"而不是"监督者选择哪个 agent 下一个？"这使系统看起来像一个 pub/sub 事件网格。

### LangGraph 的 Swarm Architecture

LangGraph 2025 文档明确将"Swarm Architecture"描述为多 agent 模式之一：agent 是节点，但边形成带循环的有向图，任何节点都可以从池中激活。工作者通过条件从可用工作中选取，而不是由监督者分配。

### 失败模式：饥饿和热点

如果所有工作者都拉取最快可用的任务，长时间运行的任务永远不会被选取，直到它们是唯一剩下的。经典队列饥饿。

缓解措施：
- 带显式老化的优先队列（随等待时间增加优先级）。
- 工作者专业化：某些工作者只接"长"任务。
- 背压：限制多少快任务进入队列。

### 基于内容的路由链接

Swarm 自然地与基于内容的路由（Lesson 22）配对。不是通用队列，而是每种消息类型一个队列。专家工作者只订阅其类型。这是扩展到数千 agent 的消息总线架构的基础。

## Build It

`code/main.py` 实现了 4 个工作者线程从共享 `queue.Queue` 拉取的 swarm。任务有可变时长（有些快，有些慢）。演示对比：

- **顺序基线：** 一个工作者串行处理所有任务。
- **固定分配：** 每个任务预分配给特定工作者（监督者风格）。
- **Swarm：** 工作者从共享队列拉取。

Swarm 自动平衡负载；固定分配在分配的任务慢时让快工作者空闲。

运行：

```
python3 code/main.py
```

输出显示每工作者任务计数（swarm 分配不均匀但最优）和挂钟时间。

## Use It

`outputs/skill-swarm-fit.md` 评估任务应该使用 swarm vs 监督者。输入：任务独立性、时长方差、排序要求、可调试性需求。

## Ship It

检查清单：

- **带老化的优先队列。** 防止长任务饥饿。
- **工作者幂等性。** 如果工作者在运行中崩溃，任务可能被拉取多次。工作者必须是幂等的。
- **持久队列。** 生产中使用 Kafka、Redis Streams 或数据库支持的队列。`queue.Queue` 仅在内存中。
- **每任务可观测性。** 每个任务有追踪 ID；每个工作者记录开始/结束。
- **背压。** 如果队列增长快于工作者排空，减慢生产者。

## 练习

1. 运行 `code/main.py`。在可变时长工作负载上 swarm 比顺序快多少？比固定分配快多少？
2. 添加优先队列变体（使用 `queue.PriorityQueue`）。按任务"重要性"字段分配优先级。观察在持续负载下低优先级任务是否会饥饿。
3. 实现热点检测器：当任何工作者处理的任务数是最慢工作者的 3 倍时记录日志。这说明了任务时长分布的什么？
4. 阅读 Matrix 论文（arXiv:2511.21686）摘要和第 3 节。识别 Matrix 接受的一个具体权衡（可扩展性收益）和它放弃的一个（可追踪性、确定性）。
5. 将 swarm 演示转换为使用 (task_type, payload) 元组的 `queue.Queue`，工作者只订阅特定类型。当任务异构时什么路由规则有意义？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|---------|---------|
| Swarm 架构 | "去中心化 agent" | 工作者从共享队列拉取；没有中央编排器。 |
| 事件总线 | "Agent 订阅主题" | 按类型或内容将任务路由到工作者的消息代理。 |
| 饥饿 | "任务永远不运行" | 低优先级任务因为更高优先级工作持续到达而永远不被选取。 |
| 热点 | "一个工作者淹没" | 一个工作者获得大多数任务的负载不平衡。 |
| 背压 | "减慢生产者" | 当队列填满时向上游发信号停止生产的机制。 |
| 幂等工作者 | "安全重跑" | 处理两次的任务产出相同结果。因为工作者可能在运行中崩溃所以必需。 |
| 持久队列 | "崩溃后存活" | 由磁盘或复制存储支持的队列；工作者崩溃时任务不丢失。 |
| Matrix 框架 | "全消息传递 swarm" | 数据和控制流都是分布式队列上的序列化消息。 |
